# MCAP Robustness, Compression, and Scale

Predecessors: `0014` (decode + `decode_status`), `0015` (typed extraction),
`0016` (attachments + metadata records)

Before this feature set the adapter had only ever met one tiny, uncompressed,
JSON-encoded synthetic fixture. The real corpora are the opposite on every axis:
compressed (didi/demo **zstd**, nuScenes **lz4**), multi-encoding (`ros1msg`,
`protobuf`, `json`, `cdr`), and large (59 MB to 7.9 GB). This makes the
workstream production-credible on those bytes.

## What changed

### Compression codecs are first-class
`zstandard` and `lz4` are now base dependencies (real logs are always
compressed). The `mcap` reader decompresses transparently; the adapter only adds
a clear failure when a codec is genuinely missing. Both codecs are exercised on
**genuine corpus bytes** via tiny sliced fixtures, not synthetic stand-ins.

### Decoders wired as extras
`pyproject.toml` exposes one extra per family — `ros1`, `ros2`, `protobuf` — plus
`all`. CI (`uv sync --extra dev`) installs all three, so the didi (ros1),
nuScenes (protobuf + json), and synthetic ros2/cdr decode paths run for real.

### A read-error taxonomy
`McapAdapter` reads now classify failure instead of collapsing everything into
"not a valid MCAP":

| Error | Meaning | Disposition |
| --- | --- | --- |
| `CodecUnavailableError` | a chunk's codec is not installed | **hard error**, names the package to install; no run written |
| `CorruptMcapError(status="crc-mismatch")` | a chunk CRC does not match its contents | readable prefix kept, run **quarantined** |
| `CorruptMcapError(status="truncated")` | the byte stream ends mid-record | readable prefix **recovered**, run **quarantined** |
| `AdapterError` | missing file / not MCAP / summary-less | hard error (unchanged) |

A missing codec is the only *fixable* failure (install the codec), so it is the
only one that aborts; damage is data we partly keep.

### Streaming, bounded-memory ingest
`ingest_mcap` no longer materializes the whole observations table. Rows stream to
the lake in batches of `batch_size` (default **1024**; drop toward **~100** for
image/lidar-heavy logs whose rows carry large `payload_blob` bytes). Batch size
is a pure performance knob — it never changes which rows land or their contents.

### CRC validation, opt-out for the hot path
Chunk CRCs are validated as messages are read (`validate_crcs=True`, the
default). Silent corruption no longer passes through. Trusted-data callers can
pass `validate_crcs=False` to skip the check on the hot path.

### Corruption is quarantined, never crashed
A CRC mismatch or truncation yields a **quarantined run with its recovered
prefix**, plus an integrity verdict the quality gate honors:

- the verdict (`integrity.status`, `integrity.recovered`, `integrity.reason`)
  travels on `runs.metadata`, where it survives the quality gate's
  `quality_flags` overwrite;
- ingest sets `quality_flags = ["quarantined", "integrity:<status>"]` immediately
  (do not silently pass);
- a new `byte-integrity` quality rule re-derives the quarantine from metadata, so
  re-validation cannot clear a damaged run.

### Per-run decode + integrity summary
`IngestReport` (and the `ingest mcap` CLI) now surface `decode_by_status`,
`decode_by_encoding`, `integrity_status`, and `recovered_count`; the ingest
`transform_runs` row records the same plus `batch_size`.

## Deterministic CI fixtures

Tiny, byte-stable fixtures — generated by checked-in `make_*` scripts — cover
every path without checking in gigabytes:

| Fixture | Source | Covers |
| --- | --- | --- |
| `slice_ros1_zstd.mcap` | sliced from `demo.mcap` via `export` | zstd codec + ros1 decode |
| `slice_mixed_lz4.mcap` | sliced from a nuScenes scene via `export` | lz4 codec + json & protobuf in one file |
| `ros2_imu.mcap` | generated with `mcap-ros2-support` | cdr / ros2msg decode |
| `crc_corrupt.mcap` | built + one chunk byte flipped | CRC-mismatch quarantine |
| `truncated.mcap` | built multi-chunk + tail chopped | prefix recovery + quarantine |

`demo.mcap` (ros1msg + zstd) stands in for didi for the zstd slice: identical
encoding + compression, far smaller and faster. The slice scripts take an
optional source path, so a maintainer can point them at a real didi file.

## Memory: documented perf check

`tests/test_perf_memory.py` is an opt-in smoke test (skipped by default; needs a
corpus file via `LANCEDB_ROBOTICS_PERF_MCAP`). It ingests at `batch_size=100` and
asserts peak Python heap stays far below the file size.

Observed:

| Corpus | File | Messages | Peak heap |
| --- | --- | --- | --- |
| didi `sitting.mcap` (ros1 + zstd) | **2766 MB** | 31,791 | **37 MB** |
| nuScenes scene (protobuf + json + lz4, image-heavy) | 435 MB | 35,967 | 54 MB |

Peak heap is governed by `batch_size`, not file size — a 2.7 GB log ingests with
the same flat ~tens-of-MB profile as a 435 MB one.

## The narrative

The lakehouse handles real, compressed, multi-gigabyte robot logs — decoding what
it can across ros1/protobuf/json/cdr, streaming rows to disk with bounded memory,
validating CRCs, quarantining what it cannot verify, and never losing the run.
